# Git Workflow

## Sources
- `specs/constitution.md` — Quality Gates, CONST-13 (secrets never committed), CONST-14 (quality gates before merge)
- Repository layout: `docs/`, `specs/`, `src/`, `tests/`, `inputs/`, `.env.example`, `pyproject.toml`
- Existing commit history: conventional commit style (feat/fix/docs/refactor/chore/test)

## Branch Naming

| Change class | Path globs that trigger it | Branch pattern | Base branch |
|--------------|---------------------------|----------------|-------------|
| Docs | `docs/**` | `docs/<slug>` | `main` |
| Specs | `specs/**` | `specs/<slug>` | `main` |
| Feature (new capability) | `src/**`, `tests/**` (new file or new function) | `feat/<slug>` | `main` |
| Fix (bug correction) | `src/**`, `tests/**` (existing code changed) | `fix/<slug>` | `main` |
| Test-only | `tests/**` only | `test/<slug>` | `main` |
| Config/build | `pyproject.toml`, `.env.example`, `.github/**` | `chore/<slug>` | `main` |
| Refactor | `src/**` (no behavior change, no new files) | `refactor/<slug>` | `main` |

`<slug>` is a kebab-case 1–5 word summary of the change (e.g. `row-execution`, `pppk-gold-set`, `pk-invariants`).

## Mixed-Change Rule
When a changeset spans multiple classes (e.g. `src/` + `docs/`), use the **highest-precedence class** in this order: feat > fix > refactor > test > chore > docs > specs. If the mix is docs+specs only, use `docs/<slug>`. Never combine a spec change with an `src/` change in the same branch without a strong reason documented in the PR.

## Commit Convention
Format: `<type>(<scope>): <subject>`
- **type**: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`
- **scope**: the module or area changed (e.g. `translate`, `harness`, `pk-service`, `models`, `cli`, `ui`)
- **subject**: imperative mood, ≤72 chars, no trailing period
- **body** (optional): why the change was needed, not what it does
- **mandatory trailer**: `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>` when generated by AI assistant

Examples:
```
feat(execute): add bounded row fetch with pagination to execute_rows

fix(translate): emit only closed-set members in translate_runtime_filter

docs(gold-set): replace sme_stage_cases.csv references with PPPK.xlsx

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

## Push & PR Policy
- Push only after the Quality Gates (all four) pass locally.
- Open a PR for every branch targeting `main`. PRs must not be merged without CI passing.
- PR title mirrors the commit message subject (same `<type>(<scope>): <subject>` format).
- PR body: summary (3 bullets max), test plan (what was run), link to any relevant docs/specs file.
- Direct push to `main` is not permitted except for hotfixes with explicit maintainer approval.
- Squash-merge preferred to keep history readable.

## Pre-Commit Gates
All four must pass before a commit is allowed:
1. `python3 -m compileall src/oppp`
2. `ruff check src tests`
3. `ruff format --check src tests`
4. `pytest -q`

If `ruff format --check` fails, run `ruff format src tests` to fix, then re-stage.

## Never Commit
- `.env` and any file matching `*.env` (real credentials)
- `credentials.json`, `*.pem`, `*.key`, `*.p12`, `*.pfx`
- `docs/PPPK.xlsx` modifications (gold set is a provided input; do not modify via git)
- `inputs/*.csv` modifications from pipeline code (CSVs are provided inputs)
- `__pycache__/` and `*.pyc` files (covered by `.gitignore`)
- Any file that `detect-secrets` flags as containing a secret pattern

## Safety & Idempotency
- **Branch already exists**: `git switch <existing-branch>` rather than re-creating it; check for conflicts with `git status` before adding new commits.
- **Nothing to commit**: run `git status`; if clean, skip the commit step — do not create an empty commit.
- **PR already open**: use `gh pr edit` to update title/body rather than opening a duplicate.
- **Force push**: never force-push to `main` or any shared branch. Force push is only permitted on personal feature branches before any reviewer has been added.
- **Amend**: amend only unpushed commits. Once pushed, create a new commit.
